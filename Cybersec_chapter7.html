// sección 03 — alertas

Cuándo debe
sonar el alarm

Alertas demasiado sensibles crean ruido. Muy permisivas dejan pasar incidentes. Estos umbrales son el punto de equilibrio para una API FastAPI en producción.

CRÍTICO
Health check fallido
GET /health != 200 durante > 2 minutos
El sistema está caído o degradado. PostgreSQL o Redis no responden. Acción inmediata requerida.
→ PagerDuty + Slack #incidents + email
CRÍTICO
Error 500 rate alta
HTTP 5xx > 5% del total de requests en 5 min
El servidor está lanzando excepciones no controladas en masa. Posible bug desplegado o DB sobrecargada.
→ Sentry captura + alerta Slack #incidents
ALTO
Brute force detectado
> 10 intentos de login fallidos desde una IP en 5 min
Posible ataque automatizado contra cuentas de usuario. El sistema de lockout ya está activo pero hay que investigar el origen.
→ Slack #security + log forense de la IP
ALTO
Latencia P95 elevada
P95 > 2000ms por más de 10 minutos
Las queries están tardando demasiado. Posible falta de índices, connection pool saturado, o carga inesperada.
→ Slack #ops + revisar slow query log
MEDIO
CPU sostenida alta
CPU > 85% por más de 15 minutos
Carga inusual. Puede ser tráfico legítimo (escalar) o un proceso en bucle infinito (investigar).
→ Slack #ops + revisar métricas de requests
MEDIO
Certificado TLS por expirar
Certificado expira en < 30 días
Let's Encrypt renueva automáticamente, pero si la renovación falla, HTTPS cae en 30 días. Verificar que certbot está corriendo.
→ Email semanal + revisar certbot
MEDIO
CVE detectado en dependencias
pip-audit encuentra vulnerabilidad HIGH o CRITICAL
Una librería tiene una vulnerabilidad conocida. Urgente si es CRITICAL, importante si es HIGH.
→ GitHub Issue + Slack #security
INFO
Deploy completado
Push a main con CI/CD exitoso
Notificación informativa. Confirma que el deploy fue exitoso y el health check está verde post-deploy.
→ Slack #deploys

// sección 04 — respuesta

Anatómía de un
incidente real

Escenario: ataque de SQL injection detectado en producción a las 03:47. Seis fases, acciones concretas en cada una.

03:47
Fase 1 — Impacto
Sentry captura 47 errores 500 en 3 minutos
Stack trace apunta a sqlalchemy.exc.ProgrammingError en el endpoint GET /api/v1/search. Sentry envía alerta instantánea a Slack #incidents. El on-call recibe notificación.
MTTD (Mean Time To Detect): 3 minutos ← Sentry funciona
03:52
Fase 2 — Evaluación
Analizar scope del incidente
Revisar logs de los últimos 10 minutos. Verificar si hay exfiltración de datos (queries SELECT masivas). Evaluar si la brecha está activa o cerrada. Determinar si escalar.
$ docker compose logs app --since="2024-04-25T03:40:00" | grep -i "sql\|error\|injection"
03:55
Fase 3 — Contención
Bloquear el vector de ataque activo
Identificar la IP atacante en los logs. Bloquear en nginx inmediatamente. Si es necesario, deshabilitar el endpoint vulnerable temporalmente mientras se aplica el parche.
$ docker compose exec nginx nginx -s reload && echo "IP bloqueada" # Opcional: deshabilitar endpoint # Comentar la ruta en router y redeploy
04:10
Fase 4 — Análisis forense
Entender exactamente qué pasó
Extraer logs completos del período. Reproducir el payload en entorno de desarrollo. Verificar el código vulnerable. Determinar si hay datos comprometidos y cuáles.
# Buscar queries anómalas en PostgreSQL $ docker compose exec db psql -U cipher_user -c "SELECT query, calls, total_exec_time FROM pg_stat_statements ORDER BY calls DESC LIMIT 20;"
05:30
Fase 5 — Resolución
Parche, tests y redeploy
Corregir el código vulnerable (usar ORM parametrizado). Escribir un test que reproduzca el ataque y verifique que el parche funciona. Desplegar con pipeline completo.
# El test que previene que vuelva a pasar: # pytest tests/security/test_sql_injection.py -v
09:00
Fase 6 — Aprendizaje
Postmortem sin blame — ¿Cómo prevenimos el próximo?
Documentar el incidente, el impacto real, las acciones tomadas y las mejoras del sistema. El postmortem no busca culpables — busca fallos en el proceso que permití que el bug llegara a producción.
MTTR (Mean Time To Recover): 1h 43min ← objetivo: < 1h

// sección 06 — procedimientos

Runbooks — qué hacer
cuando suena la alarma

Procedimientos documentados que cualquier miembro del equipo puede ejecutar a las 3am sin pensar demasiado.

R1
Caída total
API no responde (/health = timeout)
1. docker compose ps → ver estado contenedores
2. docker compose logs --tail=50 app → último error
3. Si OOM: docker compose restart app
4. Si DB caída: docker compose restart db
5. Si persiste: rollback al último deploy estable
6. Notificar en Slack #incidents con ETA
R2
Lentitud
Latencia P95 > 2s (sistema lento)
1. Verificar CPU y RAM en Render dashboard
2. pg_stat_statements → queries más lentas
3. Ver conexiones activas a PostgreSQL
4. Si connection pool lleno: reiniciar app
5. Si queries lentas: analizar con EXPLAIN ANALYZE
6. Si carga legítima: escalar workers
R3
Seguridad
Posible brecha de seguridad activa
1. NO eliminar logs — son evidencia forense
2. Bloquear IP atacante en nginx inmediatamente
3. Revocar todos los JWT activos (rotar SECRET_KEY)
4. Evaluar si hay datos comprometidos
5. Si hay PII expuesto: notificar usuarios afectados
6. Postmortem obligatorio en 24-48h
R4
Base de datos
PostgreSQL no responde o corrompido
1.
1. docker compose exec db pg_isready -U cipher_user
2. Verificar espacio en disco del volume
3. docker compose logs db → ver errores PostgreSQL
4. Restaurar desde último backup automático (Render)
5. Re-ejecutar alembic upgrade head
6. Verificar integridad de datos críticos
R5
Deploy
Deploy roto — health check falla post-deploy
1. Render hace rollback automático si health falla
2. Si no: Render Dashboard > Deploys > Rollback
3. Si migración fañó: alembic downgrade -1
4. Reproducir el fallo en staging antes de re-intentar
5. Agregar test que cubra el caso fallido
R6
Credenciales
Secret key o credencial comprometida
1. Revocar inmediatamente en el proveedor
2. Generar nuevo valor — python secrets.token_hex(32)
3. Actualizar en Render/Railway Environment
4. Reiniciar todos los servicios
5. Si SECRET_KEY: todos los JWT quedan inválidos
6. Auditar quién tenía acceso a la credencial

# app/core/logging_config.py — logging de producción con loguru
from loguru import logger
from fastapi import Request
import sys, os, json, time
 
def setup_logging():
    logger.remove()
 
    # JSON estructurado a stdout — Docker + Render lo captura
    logger.add(
        sys.stdout,
        format="{time:YYYY-MM-DD HH:mm:ss.SSS} | {level:<8} | {name}:{line} | {message}",
        level=os.getenv("LOG_LEVEL", "INFO"),
        serialize=True,           # JSON para parsear en producción
        backtrace=False,          # sin tracebacks en stderr (van a Sentry)
        diagnose=False,
    )
 
    # Archivo con rotación diaria y compresión
    logger.add(
        "/app/logs/cipher-api-{time:YYYY-MM-DD}.log",
        rotation="1 day",
        retention="30 days",
        compression="gz",
        level="WARNING",
        serialize=True,
    )
 
# Middleware que loguea cada request
class RequestLoggingMiddleware:
    async def dispatch(self, request: Request, call_next):
        t0 = time.time()
        req_id = request.headers.get("x-request-id", "-")
 
        response = await call_next(request)
        ms = round((time.time() - t0) * 1000, 2)
 
        # Determinar nivel según status
        log_fn = logger.info
        if response.status_code >= 500: log_fn = logger.error
        elif response.status_code >= 400: log_fn = logger.warning
 
        # Log estructurado — SIN PII, sin contraseñas
        log_fn("http_request",
            method=request.method,
            path=request.url.path,
            status=response.status_code,
            ms=ms,
            ip=request.client.host,
            request_id=req_id,
        )
        return response

# app/core/sentry_config.py — Sentry para error tracking
import sentry_sdk, os
from sentry_sdk.integrations.fastapi import FastApiIntegration
from sentry_sdk.integrations.sqlalchemy import SqlalchemyIntegration
from sentry_sdk.integrations.redis import RedisIntegration
from sentry_sdk.integrations.loguru import LoguruIntegration
 
def setup_sentry():
    dsn = os.getenv("SENTRY_DSN")
    if not dsn:
        return  # silencioso en desarrollo
 
    sentry_sdk.init(
        dsn=dsn,
        environment=os.getenv("ENVIRONMENT", "development"),
        release=os.getenv("GIT_SHA", "unknown"),
 
        integrations=[
            FastApiIntegration(),       # captura exceptions de FastAPI
            SqlalchemyIntegration(),    # rastrea queries lentas
            RedisIntegration(),         # errores de Redis
            LoguruIntegration(),        # integra con loguru
        ],
 
        # Performance monitoring
        traces_sample_rate=0.1,     # 10% de requests trackeados
        profiles_sample_rate=0.05,  # 5% con profiling de CPU
 
        # Privacidad — no enviar PII
        send_default_pii=False,
        before_send=scrub_sensitive_data,
 
        # Ignorar errores esperados
        ignore_errors=["HTTPException"],  # 404, 422, etc son normales
    )
 
def scrub_sensitive_data(event: dict, hint: dict) -> dict:
    # Limpiar headers sensibles antes de enviar a Sentry
    if "request" in event:
        headers = event["request"].get("headers", {})
        for k in ("Authorization", "Cookie", "X-API-Key"):
            if k in headers:
                headers[k] = "[REDACTED]"
 
    # Limpiar datos de usuario del contexto
    if "user" in event:
        # Solo mantener ID, eliminar email/nombre
        user_id = event["user"].get("id")
        event["user"] = {"id": user_id}
 
    return event
 
# Capturar manualmente eventos de seguridad
def capture_security_event(event_type: str, **kwargs):
    with sentry_sdk.push_scope() as scope:
        scope.set_tag("event_type", "security")
        scope.set_tag("security_event", event_type)
        scope.set_extra("context", kwargs)
        sentry_sdk.capture_message(
            f"Security: {event_type}",
            level="warning"
        )

# app/middleware/brute_force_protection.py
from fastapi import Request, HTTPException
from app.db.redis import get_redis
from loguru import logger
import sentry_sdk
 
MAX_ATTEMPTS   = 5
WINDOW_SECONDS = 300   # 5 minutos
LOCKOUT_TIME   = 3600  # 1 hora
 
async def check_brute_force(request: Request, identifier: str):
    redis = await get_redis()
    ip = request.client.host
 
    # Clave única por IP + endpoint
    attempt_key = f"bf:attempts:{ip}:{identifier}"
    lockout_key = f"bf:lockout:{ip}"
 
    # Verificar si está en lockout
    if await redis.exists(lockout_key):
        ttl = await redis.ttl(lockout_key)
        logger.warning("auth_blocked_lockout", ip=ip, ttl_s=ttl)
        raise HTTPException(
            429,
            f"Demasiados intentos. Espera {ttl // 60} minutos."
        )
 
    # Incrementar contador
    attempts = await redis.incr(attempt_key)
    if attempts == 1:
        await redis.expire(attempt_key, WINDOW_SECONDS)
 
    if attempts > MAX_ATTEMPTS:
        # Activar lockout de 1 hora
        await redis.setex(lockout_key, LOCKOUT_TIME, "1")
 
        # Alerta de seguridad
        logger.error("brute_force_lockout", ip=ip, attempts=attempts)
        sentry_sdk.capture_message(
            f"Brute force lockout: {ip}", level="warning"
        )
 
        raise HTTPException(429, "Cuenta temporalmente bloqueada.")
 
async def reset_attempts(ip: str, identifier: str):
    redis = await get_redis()
    await redis.delete(f"bf:attempts:{ip}:{identifier}")
 
# Uso en el router de auth:
@router.post("/login")
async def login(credentials: ..., request: Request, db=...):
    await check_brute_force(request, "login")
    user = await authenticate(credentials)
    if not user:
        raise HTTPException(401, "Credenciales inválidas")
    await reset_attempts(request.client.host, "login")
    return await create_tokens(user)

Postmortem —
sin blame, solo aprendizaje

El documento que escribes después de cada incidente grave. No busca culpables — busca cómo el sistema permitió que pasara.

📋 Plantilla de Postmortem — Incidente #001 — SQL Injection en /search 2024-04-25 · Durado: 1h 43min · Severidad: HIGH
📈 Impacto
Usuarios afectados: 0 (ataque contenido antes de exfiltración)
Disponibilidad: 100% (no hubo downtime)
Datos comprometidos: Ninguno — bcrypt protegió passwords
Duración del ataque: 3 minutos activo
MTTD: 3 min • MTTR: 1h 43min
🕐 Timeline del incidente
03:47 — Sentry detecta spike de 500s en /search
03:50 — Alerta llega a Slack, on-call responde
03:55 — IP bloqueada en nginx
04:10 — Análisis forense completo
05:00 — Parche desarrollado y testeado
05:30 — Deploy del parche, incidente cerrado
💡 Causa raíz
El endpoint GET /api/v1/search usaba una f-string directamente en text() de SQLAlchemy en lugar del ORM parametrizado. El código existía desde el commit inicial y no fue detectado en code review porque no había tests de seguridad para ese endpoint.
✅ Qué salió bien
• Sentry detectó el incidente en 3 minutos
• bcrypt evitó que las contraseñas fueran útiles
• El runbook existía y se siguió
• El bloqueo de IP fue inmediato
• No hubo downtime para usuarios legítimos
🔥 Qué salió mal
• El código vulnerable pasó code review
• No había tests de SQL injection automáticos
• pip-audit no detecta vulnerabilidades en nuestro código
• MTTR de 1h 43min — objetivo era < 1h
🔧 Acciones correctivas
• Inmediato: Parche y test de regression aplicado
• Esta semana: Revisar todos los endpoints con text() manual
• Este sprint: Añadir bandit al CI — detecta text() inseguro
• Próximo sprint: OWASP ZAP scan automático en staging
• Proceso: SQL injection checklist en code review template
ENTRADA #007 — CAPÍTULO 07 COMPLETADO

Hoy entendí que monitorear no es solo ver números — es construir el sistema que te despierta a las 3am cuando algo va mal, te dice exactamente qué pasó, y te da los pasos para arreglarlo. La diferencia entre 277 días de brecha sin detectar y 3 minutos de detección es exactamente Sentry + logs estructurados + alertas bien configuradas.

El postmortem sin blame fue la lección más importante. El bug existía en el código desde el día uno. No falló el desarrollador — falló el proceso: no había tests de seguridad, no había análisis estático con bandit. El objetivo no es no equivocarse — es construir sistemas que detecten y contengan los errores antes de que hagan daño real.
