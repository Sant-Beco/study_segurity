// sección 01 — el contrato

Responsabilidad
compartida

El error más común en cloud: asumir que el proveedor se ocupa de la seguridad. Se ocupan de su parte. La tuya sigue siendo tuya.

🏠
Físico
Data centers, hardware, switches, servidores físicos, alimentación, refrigeración. El proveedor de cloud tiene más seguridad física que cualquier empresa privada. Guardas, biometría, cámaras, jaulas. Tú nunca ves el hardware.
PROVEEDOR
🧠
Virtualización
Hipervisores, aislamiento de VMs, red virtual, almacenamiento distribuido. AWS, GCP, Azure se ocupan de que tu instancia no pueda leer la memoria del vecino. Esto es responsabilidad completa del proveedor.
PROVEEDOR
🌎
Red
VPC, subredes, security groups, NACLs, route tables. La infraestructura de red existe, pero tú la configuras. Un security group abierto a 0.0.0.0/0 en el puerto 5432 es tu error, no de AWS.
COMPARTIDA
🔒
IAM
Identidades, roles, políticas, permisos. AWS tiene el sistema, pero tú decides quién puede hacer qué. Un usuario con *:* en producción es tu decision — y tu vulnerabilidad.
TÚ
💾
Datos
Cifrado, backups, acceso, clasificación de datos, retención. Un bucket S3 público es completamente tu responsabilidad. AWS provee las herramientas de cifrado. Tú decides activarlas.
TÚ
💻
Aplicación
Código, dependencias, configuración, secrets, actualizaciones. FastAPI, PostgreSQL, Redis, tus librerías de Python. Los CVEs de tu código, los secrets en variables de entorno, el OWASP Top 10. Todo tuyo.
TÚ

// sección 02 — vectores

Las 9 amenazas
más comunes en cloud

El OWASP Cloud Top 10 revisado. Cada una tiene un ejemplo real y ocurre más de lo que crees.

CRÍTICO
Bucket S3 Público
Configuración por defecto incorrecta expone datos a todo internet. En 2023 más de 3 billones de registros expuestos por este error.
aws s3api put-bucket-acl --bucket mi-bucket --acl public-read
← Una sola línea lo compromete todo
CRÍTICO
Credenciales en Código
AWS Access Keys commiteadas a GitHub. Bots escanean GitHub 24/7. En 200 segundos de media el key es explotado para minar criptomonedas a tu costa.
AWS_ACCESS_KEY_ID=AKIA... en .env commiteado
Costo de mining: $50,000/mes en tu cuenta
CRÍTICO
IAM Overpermissioned
Roles con permisos de administrador cuando solo necesitan leer un secret. Si ese rol se compromete, el atacante tiene acceso total a toda la cuenta AWS.
Effect: Allow, Action: "*", Resource: "*"
← Admin total — nunca en producción
ALTO
Security Groups Abiertos
Puertos de base de datos, Redis, o administración expuestos a 0.0.0.0/0. Shodan indexa esto en horas. PostgreSQL público = datos comprometidos.
port 5432: 0.0.0.0/0 ← PostgreSQL público
port 6379: 0.0.0.0/0 ← Redis sin auth público
ALTO
IMDS v1 / Metadata API
En AWS, SSRF puede acceder a http://169.254.169.254/ y robar credenciales IAM temporales del servidor. IMDSv2 require un token de sesión, mitiga esto.
curl http://169.254.169.254/latest/meta-data/iam/
→ credentials del servidor en JSON
ALTO
Logs Desactivados
CloudTrail apagado = sin audit trail. Un atacante puede cubrir sus rastros borrando logs. Sin logs, no puedes reconstruir qué pasó en un incidente.
Primer acción del atacante con acceso AWS:
aws cloudtrail delete-trail --name prod
MEDIO
Snapshots Públicos
Snapshots de RDS o EBS marcados como públicos accidentalmente. Cualquiera puede crear una instancia desde ese snapshot y acceder a todos los datos.
aws rds describe-db-snapshots --include-public
→ Snapshots con datos de producción visibles
MEDIO
Secrets en Variables de Entorno
Variables de entorno visibles en logs de ECS, Lambda, o en la consola AWS. Usar Secrets Manager o Parameter Store con cifrado KMS.
DATABASE_URL=postgresql://user:pass@... en env
→ Visible en CloudWatch logs si se loguea
MEDIO
Sin MFA en Root Account
La cuenta root de AWS sin MFA es el punto de fallo más crítico. Si la contraseña del root se compromete sin MFA, el atacante controla toda la infraestructura.
Root account: usar SOLO para emergencias
MFA hardware (YubiKey) obligatorio en root

IAM:
least privilege en la nube

El principio de mínimo privilegio aplicado a cloud. Cada rol, usuario y servicio debe tener exactamente los permisos que necesita — ni uno más.

✗ Política peligrosa — Admin total
✗ NUNCA en producción

// sección 05 — házlo ahora

Securizar tu cuenta
paso a paso

En este orden. Los primeros cinco cubren el 80% del riesgo. Todos los demás cubren el 20% restante.

01
Activar MFA en la cuenta root — inmediatamente
La cuenta root es el acceso total a toda tu infraestructura AWS. Sin MFA, una contraseña comprometida = todo comprometido. Usar hardware key (YubiKey) si es posible.
AWS Console → IAM → Root user → Enable MFA
02
Eliminar Access Keys del root y del código
Nunca usar access keys del root. Crear usuarios IAM con permisos mínimos para operaciones específicas. En EC2/ECS usar IAM Roles, nunca access keys hardcodeadas.
aws iam delete-access-key --access-key-id AKIA... # Auditar: git log -S "AKIA" para buscar en historial
03
Revisar y cerrar Security Groups abiertos
Ninguna base de datos, cache, o servicio interno debe ser accesible desde 0.0.0.0/0. Solo el load balancer debe estar en la subred pública. PostgreSQL y Redis: solo desde el security group de la app.
aws ec2 describe-security-groups --query "SecurityGroups[?IpPermissions[?IpRanges[?CidrIp=='0.0.0.0/0']]]"
04
Activar CloudTrail en todas las regiones
CloudTrail registra cada llamada a la API de AWS. Sin él no puedes reconstruir qué pasó en un incidente. Activarlo en todas las regiones, no solo la principal. Logs a S3 con MFA delete.
aws cloudtrail create-trail --name cipher-audit --s3-bucket-name cipher-cloudtrail-logs --is-multi-region-trail --enable-log-file-validation
05
Bloquear acceso público a todos los buckets S3
En AWS hay una opción de bloqueo público a nivel de cuenta que aplica a todos los buckets actuales y futuros. Activarla es la acción con mayor impacto en el menor tiempo.
aws s3control put-public-access-block --account-id TU_ACCOUNT_ID --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"
06
Migrar secrets a Secrets Manager / Parameter Store
Variables de entorno para secrets es cómodo pero peligroso. AWS Secrets Manager con rotación automática es el estándar. Para Render/Railway: usar el sistema de secrets del proveedor, nunca .env en el repo.
aws secretsmanager create-secret --name "cipher/prod/app" --secret-string '{"SECRET_KEY":"...","DB_PASSWORD":"..."}'
07
Activar AWS GuardDuty (detección de amenazas)
GuardDuty analiza CloudTrail, DNS y VPC Flow Logs con machine learning para detectar actividad anómala. Encuentra mineros de crypto, accesos desde IPs maliciosas, y movimiento lateral.
aws guardduty create-detector --enable # Costo: ~$4/mes en cuenta pequeña # Ahorro potencial: miles en crypto mining bloqueado

# app/cloud/aws_secrets.py — integración con AWS Secrets Manager
import boto3, json, os
from functools import lru_cache
from loguru import logger
 
# Caché para evitar llamadas repetidas a Secrets Manager
@lru_cache(maxsize=10)
def get_secret(secret_name: str) -> dict:
    # En ECS/Lambda usa el IAM Role — sin credenciales hardcodeadas
    client = boto3.client("secretsmanager", region_name="us-east-1")
    try:
        response = client.get_secret_value(SecretId=secret_name)
        return json.loads(response["SecretString"])
    except Exception as exc:
        logger.error("secrets_manager_error", secret=secret_name, error=str(exc))
        raise
 
# app/core/config.py — usar Secrets Manager en producción
from pydantic_settings import BaseSettings
 
class Settings(BaseSettings):
    ENVIRONMENT: str = "development"
 
    @property
    def SECRET_KEY(self) -> str:
        if self.ENVIRONMENT == "production":
            # En prod: obtener de Secrets Manager
            secrets = get_secret("cipher/prod/app")
            return secrets["SECRET_KEY"]
        return os.getenv("SECRET_KEY", "dev-only-key")

# ── Publicar métricas custom en CloudWatch
cloudwatch = boto3.client("cloudwatch", region_name="us-east-1")
 
def push_metric(name: str, value: float, unit: str = "Count"):
    cloudwatch.put_metric_data(
        Namespace="CipherAPI",
        MetricData=[{
            "MetricName": name,
            "Value": value,
            "Unit": unit,
            "Dimensions": [{
                "Name": "Environment",
                "Value": "production"
            }]
        }]
    )
 
# En tu middleware FastAPI — trackear auth failures
async def dispatch(self, request, call_next):
    response = await call_next(request)
    if response.status_code == 401:
        push_metric("AuthFailures", 1)
    elif response.status_code == 429:
        push_metric("RateLimitHits", 1)
    return response

# app/cloud/storage.py — S3 con seguridad máxima para uploads
import boto3, uuid, magic
from botocore.exceptions import ClientError
from loguru import logger
 
S3_BUCKET  = "cipher-uploads-prod"
CDN_DOMAIN = "https://cdn.tudominio.com"
 
def upload_to_s3(content: bytes, filename: str, user_id: int) -> str:
    # Nombre del objeto: nunca el filename original
    object_key = f"uploads/{user_id}/{uuid.uuid4()}.jpg"
 
    s3 = boto3.client("s3")
    try:
        s3.put_object(
            Bucket=S3_BUCKET,
            Key=object_key,
            Body=content,
            ContentType="image/jpeg",
            # Nunca público directo — solo vía URL firmada
            ACL="private",
            # Encriptación server-side automática
            ServerSideEncryption="aws:kms",
            # Metadata sin PII
            Metadata={"uploaded-by": str(user_id)},
        )
        logger.info("s3_upload_ok", key=object_key, user_id=user_id)
        return object_key
    except ClientError as exc:
        logger.error("s3_upload_failed", error=str(exc))
        raise
 
def get_signed_url(object_key: str, expires_in: int = 3600) -> str:
    # URL firmada con expiración — acceso temporal sin exponer el bucket
    s3 = boto3.client("s3")
    return s3.generate_presigned_url(
        "get_object",
        Params={"Bucket": S3_BUCKET, "Key": object_key},
        ExpiresIn=expires_in  # 1 hora por defecto
    )
