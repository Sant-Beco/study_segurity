// sección 02 — tipos de cifrado

Una clave o dos claves:
el dilema fundamental

El problema clásico de criptografía: ¿cómo compartes un secreto sin que nadie lo intercepte?

Simétrico

Una sola clave

Alice y Bob comparten la misma clave. El problema: ¿cómo se la envían de forma segura la primera vez?

Alice
🔑 misma clave secreta
Bob
↓
Alice
🔒 "Hola Bob" cifrado
Bob
👁 Eve intercepta: ve texto cifrado pero sin la clave → ilegible
ALGORITMOS
AES-128 AES-256 ✓ ChaCha20 DES ✗
✓ Muy rápido — ideal para grandes volúmenes de datos
✓ AES-256 es prácticamente irrompible con hardware actual
✗ Problema de distribución de claves (key distribution problem)
✗ N usuarios necesitan N×(N-1)/2 claves diferentes

Asimétrico

Par de claves

Bob tiene clave pública (pública) y clave privada (solo él). Alice cifra con la pública, solo Bob descifra con la privada.

Bob
🔓 clave pública de Bob
Alice
Alice
🔒 mensaje cifrado con clave pública
Bob
👁 Eve intercepta: clave pública (inútil para descifrar) + texto cifrado (ilegible)
ALGORITMOS
RSA-2048 ✓ ECC / ECDSA Ed25519 ✓ Diffie-Hellman
✓ Resuelve el problema de distribución de claves
✓ Base de HTTPS, SSH, PGP, firma digital
✗ 100-1000x más lento que simétrico
✗ En la práctica: usa asimétrico para acordar clave simétrica (TLS)
◈ Intercambio Diffie-Hellman — Cómo dos personas acuerdan una clave sin nunca enviársela
ALICE
Secreto privado: a = 6
Público acordado: g=5, p=23
Calcula: A = g^a mod p
A = 5^6 mod 23 = 8
Envía 8 a Bob →
Recibe B=19 de Bob
Clave: 19^6 mod 23 = 2
g=5, p=23
(público)
⟷
Eve ve:
8 y 19
No puede
calcular 2
BOB
Secreto privado: b = 15
Público acordado: g=5, p=23
Calcula: B = g^b mod p
B = 5^15 mod 23 = 19
← Recibe A=8 de Alice
Envía 19 a Alice
Clave: 8^15 mod 23 = 2
✓ Ambos llegaron a la misma clave secreta (2) sin jamás enviársela. Eve vio los intercambios pero no puede calcularla — esto se llama el problema del logaritmo discreto.

bcrypt — el hash para contraseñas

SHA-256 es rápido — demasiado rápido para contraseñas. bcrypt está diseñado para ser lento. A propósito.

1
Input del usuario
"MiPassword123"
La contraseña en texto plano. Nunca debe guardarse así.
2
Salt aleatorio (128 bits)
$2b$12$R9h/cIPz0gi.URNNX3kh2O
Generado aleatoriamente para cada contraseña. Previene ataques de rainbow table.
SALT
3
Cost Factor (work factor)
2^12 = 4,096 iteraciones
Cuanto mayor el costo, más tiempo tarda. Eso es exactamente el objetivo.
COST = 12
4
Hash resultante (guardado en PostgreSQL)
$2b$12$R9h/cIPz0gi.URNNX3kh2OPST3/3He9U2W8H9W8Iak8bE9x5.3gXm
El hash final incluye: algoritmo + costo + salt + hash. Todo lo necesario para verificar.
Cost Factor: 12 Tiempo estimado: ~250ms por hash

Factor 12 = ~250ms = un atacante con GPU puede probar ~4,000 contraseñas/segundo.
Factor 14 = ~1s = ~1,000/seg. Factor 10 = ~60ms = ~16,000/seg. El balance: lento para atacante, rápido para usuario.

// sección 04 — protocolo TLS

TLS Handshake:
los primeros 100ms

Antes de que tu FastAPI procese un solo byte, esto sucede automáticamente en cada conexión HTTPS.

🖥 Cliente
Navegador / curl
ClientHello → TLS version, cipher suites, random A
← ServerHello Cipher suite elegida, random B
← Certificate Certificado X.509 del servidor (clave pública)
← ServerKeyExchange Parámetros Diffie-Hellman firmados
ClientKeyExchange → Pre-master secret cifrado con clave pública
⟷ Ambos derivan Session Keys AES-256 + MAC keys — simétricas, temporales
ChangeCipherSpec + Finished → Confirmación: ya usamos cifrado simétrico
✓ Canal seguro establecido — FastAPI recibe la request Todo lo anterior tomó ~50-100ms (1 RTT en TLS 1.3)
⚡ Servidor
Nginx → Uvicorn

  "username": "alice",

Los datos del usuario. NO cifrado, solo codificado. No guardes contraseñas aquí. exp = expiración obligatoria.
SIGNATURE — Verificación
HMAC-SHA256(
  base64(header) + "."
  + base64(payload),
  SECRET_KEY

La firma que tu FastAPI verifica. Si alguien modifica el payload, la firma no coincide. La SECRET_KEY nunca sale del servidor.

// sección 07 — amenazas

Ataques criptográficos
que debes conocer

01
Crítico

Rainbow Table Attack

Tablas precomputadas de millones de hashes. Buscan tu hash y encuentran la contraseña original. Funcionan contra MD5 y SHA sin salt. Por eso bcrypt agrega salt aleatorio.

→ Defensa: bcrypt/argon2 con salt aleatorio por usuario
02
Crítico

Brute Force + Credential Stuffing

Probar millones de contraseñas por segundo. Con MD5, una GPU puede probar 10 billones/seg. bcrypt reduce esto a ~4,000/seg. El rate limiting en tu API es otra capa.

→ Defensa: bcrypt cost≥12 + rate limiting + account lockout
03
Alto

JWT Algorithm Confusion

Cambiar el header del JWT de HS256 a "none" o RS256. Si tu servidor acepta cualquier algoritmo, un atacante puede forjar tokens sin conocer la clave secreta.

→ Defensa: algoritmo explícito en decode(), nunca aceptar "none"
04
Alto

Padding Oracle Attack

Explotar mensajes de error de descifrado para deducir el texto plano. Afecta a implementaciones CBC mal hechas. AES-GCM (modo autenticado) elimina este vector.

→ Defensa: usar AES-256-GCM, nunca AES-CBC sin MAC
05
Alto

Weak Secret Key (JWT)

Un SECRET_KEY corto o predecible en tus JWTs. Con HS256, si alguien obtiene el token puede hacer brute force offline de la clave. "secret" o "password" son claves inválidas.

→ Defensa: secrets.token_hex(32) — mínimo 256 bits de entropía
06
Alto

Quantum Computing (futuro)

RSA-2048 y ECC son vulnerables al algoritmo de Shor en computadoras cuánticas suficientemente potentes. NIST ya está estandarizando algoritmos post-cuánticos (CRYSTALS-Kyber).

→ Defensa futura: CRYSTALS-Kyber, CRYSTALS-Dilithium (NIST 2024)
ENTRADA #003 — CAPÍTULO 03 COMPLETADO

La criptografía es lo que convierte una contraseña en texto plano en 60 caracteres de ruido irreversible. La revelación más importante: bcrypt no es "más seguro" que SHA-256 por ser más complejo — es más seguro porque está diseñado para ser lento. Esa es la idea entera. Velocidad es el enemigo cuando proteges contraseñas.

TLS combina lo mejor de ambos mundos: RSA/ECC para el intercambio de claves (resuelve el problema de distribución), y AES-256 para el resto de la conversación (velocidad). Mis contraseñas ahora van como bcrypt a PostgreSQL, mis sesiones como JWT firmado con HS256, y toda la comunicación cifrada con TLS 1.3. El siguiente paso es entender cómo un atacante intentaría romper esto.
