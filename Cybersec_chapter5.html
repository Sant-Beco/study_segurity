// sección 01 — fundamentos

Docker: contenedores,
no máquinas virtuales

La diferencia importa. Contenedores comparten el kernel del host — son más ligeros, más rápidos, y arrancan en milisegundos.

¿Qué problema resuelve?

Tu FastAPI necesita Python 3.11, ciertas librerías, configuración específica de sistema. En otra máquina, otra versión, otro sistema — falla. Docker empaqueta todo en un contenedor portable.

Un Dockerfile describe exactamente qué hay dentro. Un docker-compose.yml orquesta múltiples contenedores juntos. Un comando levanta el stack completo.

Analogía

Piensa en Docker como un contenedor de barco. No importa qué haya dentro — fruta, electrónicos, ropa. El barco (servidor) maneja todos igual. El contenido (tu app) siempre viaja con todo lo que necesita para funcionar.

# app/models/user.py — modelo SQLAlchemy → tabla PostgreSQL from sqlalchemy import ( Column, Integer, String, Boolean, DateTime, text, Index ) from sqlalchemy.orm import DeclarativeBase from datetime import datetime, timezone class Base(DeclarativeBase): pass class User(Base): __tablename__ = "users" __table_args__ = ( Index("ix_users_email", "email", unique=True), Index("ix_users_username", "username", unique=True), {"comment": "Tabla principal de usuarios del sistema"}, ) id = Column(Integer, primary_key=True, autoincrement=True) email = Column(String(255), nullable=False) username = Column(String(50), nullable=False) # Nunca "password" — siempre "hashed_password" (Cap.03) hashed_password = Column(String(72), nullable=False) # bcrypt: $2b$12$[22chars salt][31chars hash] = 60 chars # String(72) con margen de seguridad is_active = Column(Boolean, default=True, nullable=False) is_verified = Column(Boolean, default=False, nullable=False) is_admin = Column(Boolean, default=False, nullable=False) created_at = Column( DateTime(timezone=True), server_default=text("NOW()"), nullable=False ) updated_at = Column( DateTime(timezone=True), server_default=text("NOW()"), onupdate=lambda: datetime.now(timezone.utc) ) last_login = Column(DateTime(timezone=True), nullable=True) def __repr__(self): return f"<User id={self.id} email={self.email}>" # La tabla resultante en PostgreSQL: # \d users # Column | Type | Nullable # id | integer | not null # email | character varying(255)| not null # username | character varying(50) | not null # hashed_password | character varying(72) | not null # is_active | boolean | not null # created_at | timestamptz | not null

Valida antes de llegar al servicio.""" email: EmailStr # valida formato email username: str = Field(min_length=3, max_length=50, pattern=r"^[a-zA-Z0-9_-]+$") password: str = Field(min_length=8, max_length=128) @field_validator("password") @classmethod def validate_password_strength(cls, v: str) -> str: """Contraseña debe tener mayúscula, número y símbolo.""" if not re.search(r"[A-Z]", v): raise ValueError("Necesita al menos una mayúscula") if not re.search(r"\d", v): raise ValueError("Necesita al menos un número") if not re.search(r"[!@#$%^&*]", v): raise ValueError("Necesita al menos un símbolo") return v class UserResponse(BaseModel): """Lo que devuelve la API. # app/routers/auth.py — endpoints de autenticación from fastapi import APIRouter, Depends, HTTPException, status from fastapi.security import OAuth2PasswordRequestForm from sqlalchemy.ext.asyncio import AsyncSession from slowapi import Limiter from slowapi.util import get_remote_address from datetime import timedelta from app.dependencies import get_db from app.schemas.user import UserCreate, UserResponse from app.schemas.auth import TokenResponse from app.services.user_service import UserService from app.core.security import verify_password, create_access_token from app.core.config import settings router = APIRouter(prefix="/auth", tags=["auth"]) limiter = Limiter(key_func=get_remote_address) @router.post("/register", response_model=UserResponse, status_code=201) @limiter.limit("3/hour") # anti-spam (Cap.04) async def register( user_data: UserCreate, db: AsyncSession = Depends(get_db), ): svc = UserService(db) # Verificar duplicados if await svc.get_by_email(user_data.email): raise HTTPException( status_code=status.HTTP_409_CONFLICT, detail="El email ya está registrado" ) return await svc.create_user(user_data) @router.post("/login", response_model=TokenResponse) @limiter.limit("5/minute") # brute force protection async def login( credentials: OAuth2PasswordRequestForm = Depends(), db: AsyncSession = Depends(get_db), ): svc = UserService(db) user = await svc.get_by_email(credentials.username) # Verificar user Y password (mismo tiempo para evitar timing attack) if not user or not verify_password(credentials.password, user.hashed_password): raise HTTPException( status_code=status.HTTP_401_UNAUTHORIZED, detail="Credenciales inválidas", # no revelar cuál falló headers={"WWW-Authenticate": "Bearer"}, ) expire = settings.ACCESS_TOKEN_EXPIRE_MINUTES token = create_access_token( data={"sub": str(user.id)}, expires_delta=timedelta(minutes=expire) ) return TokenResponse( access_token=token, refresh_token="...", # implementación completa en repo expires_in=expire * 60 )

// sección 06 — control de esquema

Alembic: tu DB
nunca más a mano

Cada cambio en tus modelos SQLAlchemy se convierte en una migración versionada. Como Git, pero para la base de datos.

Cuando
cambias un modelo
Añades is_verified al modelo User. Alembic detecta el cambio automáticamente comparando el modelo con el estado actual de la DB.
Comando
autogenerate
alembic revision --autogenerate -m "add is_verified to users"
Genera automáticamente el archivo de migración con el SQL para upgrade (aplicar) y downgrade (revertir).
Revisa
el SQL generado
Siempre revisa el archivo antes de aplicar. Alembic a veces necesita ajustes para casos complejos como renombrar columnas o constraints.
Aplica
upgrade head
alembic upgrade head — aplica todas las migraciones pendientes en orden. En docker-compose, esto corre automáticamente al iniciar la app.
Si algo falla
downgrade
alembic downgrade -1 revierte la última migración. alembic downgrade base vuelve al estado inicial. Git para tu DB.

vs el peligroso

Un .env comprometido = sistema comprometido. Estas son las diferencias que importan.

✗ Peligroso — errores comunes
# ❌ Secretos débiles SECRET_KEY=mysecret DB_PASSWORD=password REDIS_PASSWORD=1234 # ❌ Commitear .env a git # .gitignore SIN .env # ❌ Modo debug en producción DEBUG=True ENVIRONMENT=production # ❌ DB expuesta al internet DB_HOST=0.0.0.0 # acceso externo! # ❌ CORS abierto en producción CORS_ORIGINS=["*"]
✓ Seguro — mejores prácticas
# ✓ Secretos generados criptográficamente SECRET_KEY=a3f2b1c4d5e6f7a8b9c0d1e2f3a4b5c6... DB_PASSWORD=Xk9#mP2$nL7@qR4! REDIS_PASSWORD=Yt6^bN3*wQ8&vJ5~ # ✓ .gitignore incluye .env .env en .gitignore ← siempre # ✓ Debug desactivado en producción DEBUG=False ENVIRONMENT=production # ✓ DB solo accesible internamente DB_HOST=db # nombre del container # ✓ CORS específico CORS_ORIGINS=["https://tudominio.com"]
REGLA DE ORO: El .env.example se commitea a Git con todos los keys pero SIN valores. El .env real nunca toca Git. En producción (Render/Railway/DigitalOcean), configuras las variables directamente en el panel del proveedor — sin archivos .env en el servidor.

ENTRADA #005 — CAPÍTULO 05 COMPLETADO

Hoy el stack dejó de ser teoría. Con un solo docker compose up -d, FastAPI, PostgreSQL y Redis están corriendo, hablándose por una red privada interna, con datos que persisten en volúmenes y health checks que confirman que todo está vivo. Lo más revelador: los puertos 5432 y 6379 solo son accesibles desde mi máquina — desde internet, no existen.

Poetry lockea las versiones exactas. Alembic versiona el esquema de la DB como Git versiona el código. Pydantic valida que ningún input malformado llegue al servicio. La contraseña que el usuario escribe nunca se guarda — lo que llega a PostgreSQL es el hash bcrypt de 60 caracteres. La base está. Ahora toca iterar.
